# AD-SDLC v0.1 Worker Pilot Evaluation Report

This report documents the Phase-2 evaluation of the worker-stage pilot that
routes the `worker` agent through `ExecutionAdapter` (Claude Agent SDK) when
the `AD_SDLC_USE_SDK_FOR_WORKER` flag is on, otherwise through the legacy
`AgentBridge` path. The pilot landed across PRs #810 to #815 on `develop`
and is the gating evidence for the Phase Gate P2 -> P3 decision called out in
ARCH-RFC-001 section 10.

All numbers in this report are derived from static inspection of the source
tree at the report-writing checkout. Items that require runtime measurement
are explicitly labelled "Not measured" with a rationale; the recommendation in
section 6 takes those gaps into account.

## 1. Methodology

### 1.1 Comparison scenarios (5 types)

The pilot defines five integration scenarios that pair the bridge path against
the adapter path. They live in `tests/integration/worker-pilot/` and share
helpers from `_helpers.ts`:

| Scenario ID | File                         | Description                                                            | Test cases |
| ----------- | ---------------------------- | ---------------------------------------------------------------------- | ---------- |
| T1          | `simple-bug-fix.test.ts`     | Single-file bug fix (one artifact)                                     | 3          |
| T2          | `new-feature.test.ts`        | New source plus colocated unit test (two artifacts, order-independent) | 3          |
| T3          | `refactor.test.ts`           | Multi-file refactor: rename across 5 files                             | 3          |
| T4          | `with-deps-conflict.test.ts` | Verbatim `priorOutputs` propagation, failed-stage exclusion            | 4          |
| T5          | `large-issue.test.ts`        | `maxTurns` exhaustion (>10 steps) failure path                         | 4          |

Total integration cases: 17 across the five scenario files. The base wiring
test `worker-pilot.integration.test.ts` adds 8 more, for a directory total of
25 integration test cases (981 lines).

### 1.2 Equivalence axis

Each scenario asserts equivalence on a specific axis rather than byte-equal
output, because the two adapters legitimately differ in execution detail:

| Scenario | Equivalence axis                                                                               | Source                                   |
| -------- | ---------------------------------------------------------------------------------------------- | ---------------------------------------- |
| T1       | Single artifact path string is identical                                                       | `simple-bug-fix.test.ts:82-84`           |
| T2       | Artifact set equality, order ignored                                                           | `new-feature.test.ts:73` (reverse order) |
| T3       | 5-file artifact set equality, order ignored                                                    | `refactor.test.ts:90`                    |
| T4       | `priorOutputs` map equality (verbatim, completed-only)                                         | `with-deps-conflict.test.ts:115-123`     |
| T5       | `EXEC-005`-shaped failure surfaced identically (rejected promise, message contains step count) | `large-issue.test.ts:90,114`             |

The shared helper `assertEquivalentArtifacts` in `_helpers.ts:142-149` sorts
both lists by `path` and asserts deep equality, so order-insensitivity is the
default contract.

### 3.3 Execution time (p50 / p95)

Not measured. The harness needed to compute p50 / p95 over the five
scenarios is not present in this checkout: there is no `tests/performance/`
suite that targets the worker pilot specifically (`tests/performance/`
contains memory and scalability suites that target `WorkerAgent`, not the
ExecutionAdapter route). Two independent constraints apply:

1. The bridge path produces results synchronously from a stub
   (`TestOrchestrator.executeViaBridge`, `_helpers.ts:64-74`) so any
   wall-clock comparison would be dominated by harness noise rather than the
   adapter being exercised.
2. The adapter path's true latency is bounded by the live SDK round-trip,
   which requires `ANTHROPIC_API_KEY` and is gated out of CI per
   `_helpers.ts:187-190`. The mock-vs-mock comparison cannot stand in for
   this measurement because the mock adapter resolves a `Promise.resolve`
   immediately.

Inference forbidden. Per the issue's "no speculation without numbers" rule,
this row is left blank rather than estimated.

## 4. Hook firing statistics

### 4.1 PostToolUse hook coverage

The hook pipeline (`src/execution/hooks.ts`, 214 lines) registers exactly one
`PostToolUse` matcher today:

| Hook event    | Tools matched     | Callback                                      | Source                |
| ------------- | ----------------- | --------------------------------------------- | --------------------- |
| `PostToolUse` | `Edit\|Write`     | `captureEditOrWrite` -> `sink.recordArtifact` | `hooks.ts:108-160`    |
| `PreToolUse`  | none (TODO AD-P3) | no-op (key omitted from returned object)      | `hooks.ts:91,154-159` |
| `Stop`        | none (TODO AD-P3) | no-op (key omitted from returned object)      | `hooks.ts:93,154-159` |

The matcher is the regex `Edit|Write` joined from the constant
`ARTIFACT_TOOL_NAMES = ['Edit', 'Write'] as const` (`hooks.ts:109`). Per the
SDK contract documented in the hook module, the matcher list is empty for
`PreToolUse` and `Stop`, so the SDK skips those events entirely - no spurious
fires. This is asserted by the 10 unit cases in `tests/execution/hooks.test.ts`
(279 lines), which exercise the matcher, missing-`file_path` rejection, sink
failure propagation, and the `EXEC-101` / `EXEC-102` / `EXEC-103` error
branches.

### 4.2 Scratchpad registration accuracy

The pilot intentionally short-circuits scratchpad persistence: in
`AdsdlcOrchestratorAgent.createExecutionAdapter` (`AdsdlcOrchestratorAgent.ts:991-1005`)
the `ArtifactSink` is wired to a logger-only sink that records each capture at
`debug` level but does NOT persist to the scratchpad backend. The TODO is
explicit and tracked separately for P3:

| Property                                    | Pilot value (`AdsdlcOrchestratorAgent.ts:991-1005`) | P3 target                |
| ------------------------------------------- | --------------------------------------------------- | ------------------------ |
| `recordArtifact` destination                | `getLogger().debug` only                            | Scratchpad shim          |
| Idempotency                                 | not exercised                                       | Required (`hooks.ts:38`) |
| Cross-stage availability via `priorOutputs` | not exercised in pilot                              | Required for AD-09       |

This means scratchpad-registration-accuracy as a runtime metric is "not
measured" in P2. The contract that makes the future measurement meaningful
(`ArtifactSink.recordArtifact` is awaited and a rejection aborts the stage,
`hooks.ts:201-202`) is verified by unit tests, but no integration test
asserts end-to-end scratchpad presence yet.

## 6. Recommendation

### 6.1 Verdict

Conditional Proceed to P3. The pilot satisfies the P2 acceptance bar on
correctness (5/5 scenarios reach equivalence on their declared axes; 25
integration cases plus 30 execution-module unit cases hold), but the report's
section 3.3 and 4.2 metrics are "not measured" in this checkout. P3 must not
start its own work until the conditions in 6.2 are closed; otherwise we lose
the ability to compare P3 performance against a P2 baseline.

### 6.2 Conditions for unconditional Proceed

| #   | Condition                                                                                                                                                                                                       | Owner              | Closes finding |
| --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------ | -------------- |
| C-1 | Run the integration suite in a lane with `ANTHROPIC_API_KEY` exported and record observed `tokenUsage` and `toolCallCount` per scenario; replace the fixture rows in section 3.1 / 3.2 with the observed values | architecture-owner | 5.6            |
| C-2 | Add a worker-pilot performance harness under `tests/performance/` that times the adapter path against a recorded-fixture bridge baseline; report p50 / p95 over >=20 runs                                       | architecture-owner | 5.6            |
| C-3 | Replace the logger-only sink in `createExecutionAdapter` with a scratchpad shim and add an integration test that asserts the recorded artifact survives into the next stage's `priorOutputs`                    | architecture-owner | 5.4            |
| C-4 | Either emit `EXEC-005` from `SdkExecutionAdapter` on `maxTurns` exhaustion, or remove the synthetic code from `large-issue.test.ts` and reuse `EXEC-003`                                                        | architecture-owner | 5.5            |

### 6.3 What is approved unconditionally

- The routing branch in `AdsdlcOrchestratorAgent.invokeAgent` (one-line gate on
  `useSdkForWorker()` and `agentType === 'worker'`) is regression-zero for
  every non-pilot call site.
- The `HookPipeline` shape (`PreToolUse`, `PostToolUse`, `Stop`) is forward
  compatible with P3's policy/telemetry expansions: omitting an event leaves
  the SDK skipping it, so adding new matchers later is additive.
- The `MockExecutionAdapter` / `SdkExecutionAdapter` contract is the right
  abstraction line for the P3 fan-out (other stages can be migrated using the
  same pattern without touching the orchestrator's public API).

### 6.4 What is rejected for P3 entry

Nothing. P2 is intentionally narrow (one stage, one tool matcher, no
scratchpad persistence). Section 5 documents the deliberate cuts; none of them
block the next phase as long as the conditions in 6.2 are tracked.
